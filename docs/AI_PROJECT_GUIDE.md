# 使用 AI 创建项目，并减少页面的“AI 味”

## 核心原则

AI 适合加速调研、搭建、实现和检查，但不能代替产品判断。项目质量取决于你是否提前定义了目标、真实内容、设计边界和验收标准，而不是生成了多少代码。

一个可靠的顺序是：

```text
真实问题 -> 产品结构 -> 内容证据 -> 视觉规则 -> 页面实现 -> 测试与复核
```

不要从“帮我做一个漂亮网站”开始。这样的输入通常只会得到大标题、渐变背景、卡片墙和通用文案。

## 0. 步骤
1. 产品
   1. 以顶级产品经理视角执行（20年经验，Apple / Google / Meta）采用逆向工作法，先产出约1000字的 PRFAQ穿透用户表述，补齐显性与隐性需求，形成完整需求清单：功能、用户角色、使用场景、核心路径、边界与异常、数据结构输出结构化 PRD：目标用户、功能列表、MoSCoW 优先级、信息架构草图（站点地图 + 任务流）
2. 设计
   1. 设计灵感采样（转译而非模仿）
   从灵感池中随机选取2位艺术家 / 建筑师 / 工业设计师仅借鉴气质与方法，禁止临摹具体作品

   每位灵感需给出「灵感 → 网页实现」的转译说明

   灵感来源池Saul Bass, Josef Müller-Brockmann, Massimo Vignelli, Piet Mondrian,Matisse, Rothko, Dalí, Yayoi Kusama, John Maeda, Refik Anadol,Zaha Hadid, Tadao Ando, Dieter Rams, Jony Ive, Naoto Fukasawa,Susan Kare, Teenage Engineering 等强制设计约束（必须遵守）

   版式：非对称分栏、超大标题、网格秩序 + 破格

   色彩：高对比撞色、三原色几何、工业感或渐变

   形态：曲线切割、体块叠合、模块化卡片、细微纹理

   动效：200–300ms过渡，支持 prefers-reduced-motion

   语义：极简符号、变量字体轴、小幅动画、等宽数字

   禁止复刻原作构图、配色、字体、标语或布局组合

   2. 输出交互与视觉方案

   页面结构、组件清单、状态说明（默认/悬停/激活/禁用/错误/空态/加载）、可访问性、动效规范、响应式断点策略

   3. 设计系统输出 Design Tokens（色彩、字体、栅格、间距），并文字描述2–3个关键页面线框
3. 开发
   1. 语义结构（header/main/section/nav/footer），标题层级正确Google Fonts CDN（如 Inter），提供中英混排字体栈

  图标使用FontAwesome CDN 或SVG图标库（Lucide / Heroicons / Tabler）三断点响应式；

  点击区 ≥44×44px；表格与图表适配小屏图片必须提供真实链接（如 https://picsum.photos）必备 meta：viewport、color-scheme

  2. 风格偏好
  避免常见紫色或纯蓝主色，优先高识别度中性色或高质量品牌色
4. 图片
   - 图标：使用 Iconify 图标库（https://iconify.design）
   - 占位图：使用 Picsum Photos（https://picsum.photos）
   - 真实图片：使用 Pexels 搜索（https://www.pexels.com）
   - 插画：使用 unDraw（https://undraw.co）

## 1. 先建立 AI 可理解的项目上下文

在写代码前，准备一组短而明确的文档：

- **PRD**：目标用户、核心任务、页面范围、非目标和优先级。
- **Design Spec**：色彩、排版、密度、间距、组件、响应式和禁止项。
- **Technical Architecture**：技术栈、目录、数据来源、路由、测试和部署方式。
- **真实资料**：业务文档、现有代码、项目数据、截图、简历和可验证链接。

要求 AI 先读取这些资料并总结事实边界。无法确认的数字、职责和技术细节不能自行补全。真实内容是减少“AI 味”最有效的材料。

## 2. 让 AI 分阶段交付

### 先搭框架

先完成路由、布局、设计 Token、内容模型、国际化、测试框架和构建流程。不要在基础结构尚未稳定时批量生成页面。

### 先做一个代表页面

选择最能体现产品气质的页面作为视觉基准，例如首页或核心工作台。确认信息密度、字体、色彩和交互方向后，再复用到其他页面。

### 内容先于装饰

先写清楚用户要看什么、按什么顺序看、看完能做什么，再决定图片、动画和视觉效果。每个区块都应承担信息或操作职责。

### 使用可复用结构

重复页面应由数据和模板生成。例如项目案例统一使用同一个 Schema 和详情模板，内容放在 MDX、JSON 或类型化数据中，不散落在 JSX 里。

### 小步检查

每完成一类能力就运行类型检查、Lint、单元测试和浏览器检查。不要等所有页面生成完后，再一次性处理结构、样式和移动端问题。

## 3. 一个有效的 AI 任务应该包含什么

每次给 AI 的任务至少包含四部分：

1. **目标**：这次要完成什么用户结果。
2. **上下文**：需要阅读哪些文件、数据或现有页面。
3. **约束**：哪些内容不能猜、哪些设计不能使用、必须遵循什么现有模式。
4. **完成标准**：页面、状态、响应式、测试和构建需要达到什么结果。

示例：

```text
读取 PRD、Design Spec 和三个真实项目目录，完成双语 Projects 页面。
使用现有深色设计系统，采用紧凑分组列表，不做筛选和卡片墙。
项目事实只能来自提供的资料，外链必须新窗口打开。
完成后检查桌面与移动端、键盘操作、类型检查和 E2E。
```

这种任务比“做一个高级项目页”更容易得到稳定结果。

## 4. 页面为什么会有“AI 味”

常见原因不是 AI 本身，而是缺少业务约束：

- 用超大标题和大段留白代替信息层级。
- 米色、紫蓝渐变、发光球体和模糊背景被当成默认高级感。
- 所有内容都被放进圆角卡片，页面没有主次和节奏。
- 文案充满“赋能、无缝、创新、下一代”等抽象表达。
- 使用虚构数据、占位项目、无关插画或过度裁切的图库图片。
- 每个页面独立生成，导致字体、间距、组件和交互不一致。
- 动画很多，但没有帮助用户理解状态或完成任务。

## 5. 如何减少“AI 味”

### 让设计服从产品类型

作品集、后台、商城和游戏不应该使用同一种模板。先确定用户任务和使用频率，再选择内容密度和视觉表达。专业工具通常应安静、紧凑、便于扫描；品牌首页可以更有表现力，但仍要快速暴露真实内容。

### 使用真实且具体的内容

项目名称、职责边界、系统流程、技术决策、结果和链接越具体，页面越不像模板。不要把技术栈当作内容本身，要解释技术如何支持业务流程。

### 控制字体和留白

- 大标题只用于真正的 Hero，正文页和工具页使用克制字号。
- 留白用于分组，不用于制造“高级感”。
- 使用稳定的内容宽度、网格和间距 Token。
- 长标题、中文、移动端和最差数据都必须测试。

### 少用卡片，多用结构

优先使用分区、列表、表格、时间线、分栏和细线边界。只有重复对象、弹窗和真正需要独立容器的工具才使用卡片。不要在卡片里继续嵌套卡片。

### 建立有限的视觉语言

确定一个中性色基础、一到三个信号色、统一圆角、边框和阴影规则。不同区域可以通过明暗、纯色或克制的线性渐变区分，但不能每个区块都换一套视觉效果。

### 图片必须提供证据

优先使用真实产品截图、项目资产和可验证的业务场景。图库图片只能补充场景，不能伪装成项目成果。所有图片要有明确焦点、正确比例、alt 和来源信息。

### 文案像工程师说话

使用短句、事实和明确动词：构建了什么、负责什么、为什么这样设计、结果如何。删除无法验证的形容词和没有信息量的营销语。

### 动画只表达变化

动画用于 Header 状态、页面进入、展开、反馈和数据变化。保持短、可关闭，并支持 `prefers-reduced-motion`。没有状态变化时，不需要让元素持续漂浮。

## 6. 把 AI 当成不同角色使用

同一个 AI 不应只负责“生成代码”，还可以依次承担：

- **研究员**：读取资料并提取事实、风险和缺失信息。
- **产品经理**：整理用户路径、内容优先级和非目标。
- **设计评审**：检查模板感、信息密度、一致性和可访问性。
- **工程师**：按现有架构实现，并补齐错误、加载和响应式状态。
- **测试人员**：运行单元、E2E、链接、键盘、移动端和性能检查。
- **代码审查者**：寻找虚构内容、死代码、占位文案和行为回归。

角色可以切换，但每个阶段都要有明确输入和输出。

## 7. 发布前质量门禁

至少检查以下内容：

- 所有页面使用真实内容，不存在 placeholder、TODO 或虚构指标。
- 中英文内容、路由和下载文件正确对应。
- 外链行为一致，内部链接不会误开新窗口。
- 桌面和移动端没有横向溢出、文本重叠和异常留白。
- 键盘焦点可见，语义标签、alt 和颜色对比合理。
- TypeScript、Lint、单元测试、E2E 和生产构建通过。
- Metadata、Open Graph、canonical、hreflang、sitemap、robots 和 404 完整。
- 使用生产构建运行 Lighthouse，而不是用开发服务器的分数做结论。
- 人工查看关键页面截图；自动测试不能判断页面是否“看起来廉价”。

## 最终检查问题

发布前问自己：

1. 去掉渐变、阴影和动画后，内容结构还成立吗？
2. 页面是否能让目标用户在几分钟内完成核心判断或任务？
3. 每个区块是否都有真实信息或明确操作？
4. 设计是否属于这个产品，而不是属于某个流行模板？
5. AI 生成的每个事实、链接和交互是否都经过验证？

如果这五个问题都能明确回答，AI 就是在放大你的产品判断，而不是替你制造一个看似完整的模板。
