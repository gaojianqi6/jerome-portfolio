## 项目背景

RateEverything 从一个简单问题出发：电影、电视剧、音乐、书籍和播客为什么必须散落在不同的评分历史里？这个产品尝试把多个媒体类别放进同一账户和发现模型。

## 我的职责

这是我独立设计和开发的项目，包括 Next.js 用户端、React 管理后台、NestJS 产品 API、FastAPI 管理 API、共享 PostgreSQL Schema、测试、CI 和部署。

## 服务边界

NestJS 服务负责主要产品契约，并使用 Prisma 访问关系数据。FastAPI 服务通过 SQLModel 为管理工作流提供小而明确的 Python 边界。两个服务共享数据库，因此 Schema 变化和数据所有权必须清楚；只有当职责清晰度高于协调成本时，第二个服务才有意义。

## 质量与交付

Jest 覆盖服务逻辑，Cypress 覆盖关键浏览器路径。公开网站部署在 Vercel，`rating.ospreypulse.com/api` 下的请求被代理到 GCP Cloud Run 上的 NestJS API，PostgreSQL 独立托管。稳定公开路径避免把 Cloud Provider URL 暴露为客户端契约。

## 下一步加强方向

下一步是增加 Prisma 与 SQLModel 之间的 Schema Compatibility Check、更完整的审核 Audit Event，以及覆盖 Proxy 边界的 Contract Test。推荐功能成为主要卖点前，也需要先建立更清晰的推荐效果评估方法。
