## 产品背景

玄极是一套智能投顾产品。用户从开户进入风险测评，获得建议资产配置，查看组合表现，完成身份步骤，并发起再平衡等投资操作。金融信息需要尽量清楚，但限制状态、处理中、失败与确认也必须明确，不能用模糊界面掩盖风险。

产品最终服务的不止一个展示层。React 负责浏览器体验，React Native 负责移动 App，合作银行则可以使用自己的界面接入相同产品能力。这意味着后端契约不是某个页面的内部实现，而是需要长期兼容的平台边界。

## 我的职责与边界

我参与 Node.js 与 Express 服务、MySQL 业务数据、React Web、React Native、Redux 架构和原生集成点，并与产品、设计、iOS/Android、QA、基础设施、算法和银行对接团队协作。

专业算法团队负责投资数学模型。我负责把特定算法结果安全接入产品，而不是声称由我开发模型本身。绝大多数普通请求由 Express 与 MySQL 直接处理，算法服务只保留在少量清晰的专业计算边界。

## 用户流程如何贯穿全栈

### 开户与风险测评

客户端通过分步流程收集账户与适当性信息，提供校验和可恢复状态。Express 在服务端再次校验，读取 MySQL 产品数据，并返回稳定的风险档案结果。服务端始终是事实来源；客户端校验改善体验，但不会决定金融正确性。

### 投资组合与资产配置

React Selector 把 API 数据转化为面向界面的 View Model，用于配置图表、收益摘要和状态文案。UI 不需要理解完整数据库结构。特定的组合构建或调整经过算法 Adapter，之后仍回到 Web、移动端与银行共用的产品契约。

### KYC 与受保护操作

移动流程与原生工程师协作接入相机、Biometric Login 和 Push Notification。共享层负责工作流状态、校验、API Model 和错误解释；Native Module 负责设备权限与安全能力。敏感提交需要防重复操作、服务端校验和清晰审计记录。

## “80% 复用”具体指什么

这里并不是 80% UI 完全相同。浏览器路由、表格、Hover、移动导航、触控和原生权限本来就是不同问题。真正共享的是在所有平台都应该保持同一语义的部分：

- Redux Actions、Reducers 与 Selectors。
- API Request / Response Models。
- 校验规则和工作流决策。
- 登录状态和错误解释。
- 投资组合数据转换与状态逻辑。

React Web 与 React Native 在公共核心外使用各自适合的平台组件。这样既保持产品行为一致，也不会让浏览器组件假装成原生页面。

## Express 与 MySQL 的职责

Express 作为 API/BFF 边界，鉴别自有或合作方调用者，校验输入，执行产品规则，访问 MySQL Repository，转换外部故障，并返回稳定响应。用户、账户、产品、风险档案、组合版本和订单记录之间存在清晰关系，因此 MySQL 比客户端状态更适合作为持久事实来源。

对于写操作，重点是事务、幂等和兼容。禁用按钮只能改善 UX，真正防止重复金融操作仍需要服务端唯一约束和事务。API 变化也必须兼容无法立即升级的移动客户端，以及需要约定迁移窗口的银行系统。

## 关键技术决策

**共享行为，不强行共享渲染。** 公共 Redux 与领域包保护一致性，平台专属 UI 保证可用性和性能。

**把算法隐藏在 Adapter 后面。** 产品 API 负责鉴权、适当性上下文、持久化、错误和 Response Shape；专业服务只提供特定计算。

**从一开始考虑多个客户端。** 银行控制自己的体验，因此版本化契约、合作方身份、Scope、审计和兼容变化都是必要条件。

**金融事实留在服务端。** 客户端计算可以辅助展示，但组合、KYC 和操作状态必须以后端服务和 MySQL 为准。

## 工程视角

原来的架构选择依然成立：共享领域行为、平台原生展示，以及能够服务自有代码之外客户端的稳定后端契约。在现代实现中，同样的边界可以自然对应到生成式 TypeScript Contract、RFC 9457 Problem Details、显式 Idempotency Key、MySQL Transaction Integration Test，以及贯穿 Express、存储、缓存与算法调用的 Telemetry。
