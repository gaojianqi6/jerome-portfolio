## 项目背景

Carsome 不是一个单独的网站。用户看到的交易平台、内部员工使用的运营系统，以及仓库和物流依赖的库存服务，都在描述同一辆车生命周期的不同部分。一条卖车线索可能继续进入检测、上架、仓库活动、物流任务和交易流程。真正的工程问题，是让这些视角保持一致，又不让每个团队的每次发布都彼此阻塞。

## 我的职责与贡献边界

我的职位名称是 Frontend Engineer，但实际工作越过了前端边界。我开发和维护 Nuxt 官网，主导 CMS 的共享前端结构，也直接参与 NestJS Inventory Service。产品、设计、运营、后端、移动端和基础设施同事各自负责相应领域；我的主要贡献是 Web 交付、CMS 共享基础、库存工作流，以及连接这些部分的接口契约。

我还接触和维护过既有的 PHP CodeIgniter CMS。它长期承载线索、检测、定价、竞拍、报表、权限与外部集成。这里的目标不是为了换技术而重写，而是在保留有效业务流程的同时，逐步引入新模块和更清晰的服务边界。

## 三个系统，一条运营链路

### 面向客户的官网

Nuxt.js 官网通过 SSR 支持可索引的车辆页和活动页，并提供车辆卡片、图库、估价与预约等响应式组件。图片采用延迟加载，布局同时考虑宽屏桌面与移动端流量。这一层把复杂车辆信息转化成买家或卖家能够采取行动的界面。

### CMS 与微前端

CMS 在渐进迁移阶段同时存在 React、Vue 3 和 AngularJS 模块。Lerna Monorepo 提供共享的 TypeScript 类型、API Client、UI 组件、Hooks、Lint、Format 和提交检查。每个模块保留独立构建和发布能力，但对运营人员仍表现为一个统一工作台。

### Inventory Service

NestJS 服务把库存理解为工作流，而不是一张 CRUD 表。它处理入库、验证、运输、调拨、出库、维修、报废、盘点、二维码生成与验证、任务分配和通知。Controller 保持在传输边界，Service 负责业务状态迁移，并协调持久化和外部副作用。

## 数据与并发设计

MongoDB 符合主要读取模式：运营人员通常需要知道一辆车现在在哪里、之前发生过什么、下一步是什么。Car Document 保存 `carInfo`、`activities`、`tasks`、分配信息、日志，以及 `lastActivity` / `lastTask` 状态快照。最新状态字段让仓库列表读取更快，数组则保留完整时间线。

这个选择把更多责任放到了写入路径。服务需要同时维护历史与当前状态，阻止非法任务迁移，并处理并发修改。二维码操作使用带 TTL 的 Redis Key 阻止短时间重复扫描；异步任务可以携带稳定 Job ID，让重试不会悄悄制造重复业务动作。

## 服务之间如何通知

并不是每个下游更新都应该让用户等待。ERP 回调和交易状态更新进入 Redis 支撑的 Bull Queue，由 Worker 执行集成、记录失败，并在需要时保存失败任务信息供运营排查。Transfer Service 通过统一 Adapter 调用独立的 Go Transport API，包含内部鉴权和错误映射。

这种拆分让仓库操作保持响应，也让依赖故障更容易隔离。它同时暴露出下一步改进方向：HTTP 与 Queue Job 之间贯通 Correlation ID、明确的 Dead Letter 和 Replay 工具、更严格的幂等契约，以及 Queue Age 告警。

## 关键技术决策

**演进，而不是强行重写。** 多框架和 PHP CMS 是现实约束，不是为了展示技术数量。共享契约和独立构建让迁移能够继续，同时不冻结业务交付。

**以车辆作为聚合根。** 活动信息丰富的 MongoDB Document 能同时提供当前状态和历史。若时间线增长到不适合继续嵌入，可以归档旧活动，而不改变主要运营读取模型。

**把外部副作用移出请求主链路。** Redis 与 Bull 让 ERP 和交易更新可以重试、记录和观察，同时不让一线运营人员等待所有外部依赖。

**把内部运营也当成产品。** 校验、权限、健康检查、日志、失败任务可见性和稳定错误映射不是上线后的修饰，而是工作流的一部分。

## 现在会如何加强

我会补充贯穿 Gateway、API、MongoDB、Queue 与 Transfer Call 的 OpenTelemetry；拆分 Liveness 与 Readiness；根据真实查询模式记录并验证组合索引；为失败任务提供有责任人和审计记录的 Replay 工具。核心方向仍然有效：清晰的库存领域、轻薄的传输层、异步副作用和稳定客户端契约。
