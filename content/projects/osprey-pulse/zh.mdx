## 产品方向

Osprey Pulse 最初围绕体育数据、预测与社区评分展开。这个方向在技术上有探索价值，但没有形成足够强的持续使用理由。我把产品重新定位到一个更直接的承诺：**每天让一个重要目标向前一点**。

现在的体验从用户用自己的话写下目标开始。AI 帮助把它整理成结构化计划，但每一部分都可以由用户检查和编辑。Today 提供当天的一小步；完成任务或打卡会积累进度；Journey 保存长期记录；用户的宠物伙伴动动也会随这些努力成长。宠物不是流程完成后附加的装饰，而是持续行动的反馈界面。

首版刻意保持聚焦。一个进行中的目标比“管理整个人生”的 Dashboard 更容易理解；漏掉一天也不会惩罚用户或让宠物降级。好友和更丰富的分享已经进入产品方向，但当前上线核心仍是目标、计划、每日进度、私人历史和动动成长循环。

## 从 AI 回答到持久计划

AI 边界返回结构化计划数据，而不是让产品其他部分继续解析自然语言。Provider Route 管理模型、凭据、每分钟请求、每日用量、重试与冷却规则。Draft 可以先询问缺少的信息；只有用户检查确认后，计划才会成为持久数据。

AI 失败被当作正常产品状态。无效 JSON、Rate Limit、Route 用尽或缺少 Provider 配置时，可以退回到服务端校验过的模板。这样既保证首次使用流程可用，也避免外部模型成为系统事实来源。

## 契约、状态与历史

C# API 发布 OpenAPI，前端 Monorepo 从同一来源生成 TypeScript 类型、Zod Schema 和 SDK，不维护另一套手写 DTO。ASP.NET Core 返回 RFC 9457 Problem Details，让 Web 与 Native 用一致方式展示字段错误和恢复操作。

数据模型明确区分“可编辑的未来”和“不可改写的过去”。计划可以变化，但已完成日期保留当时的任务快照。XP 通过 Append-only Event 变化；任务完成、打卡和宠物互动使用 Idempotency Key，网络重试不会悄悄重复奖励进度。

## 可迁移的生产链路

公开前端运行在 Vercel。API 请求先到 Cloudflare Edge，再经 Cloudflare Tunnel 进入 Azure VM 上仅监听 Loopback 的 Nginx，最终到达 Docker 化的 ASP.NET Core 服务。VM 不开放公网应用入站端口，Azure PostgreSQL 也只使用私网访问。

整个平台可以重复创建。OpenTofu 管理 Azure 与 Cloudflare 资源、DNS、Key Vault、Tunnel 和 R2 Bucket；Ansible 准备主机并部署 Compose Stack；GitHub Actions 构建 amd64 Image，以不可变 Commit SHA 推送 GHCR，通过 Cloudflare Access 发布，并 Smoke Test 公开 Health Endpoint。Docker Health Check、Autoheal、日志轮转与 Nginx Retry 让这台刻意保持轻量的 VM 仍能稳定运行。

这个结构也没有把客户端锁死在某个 Compute Provider 上。API Hostname 保持稳定；免费 VM 不再合适时，Docker、Nginx、Tunnel、Ansible 与 Registry 可以一起迁移，Web 和 Mobile 不需要更换 Provider URL。

## 当前已经上线的内容

Marketing Website 与 Web 体验已在 `ospreypulse.com` 上线；下载页通过 `downloads.ospreypulse.com` 提供由 Cloudflare R2 承载的 Android 预览 APK。生产 API 暴露公开 Health Path，应用数据仍留在私有基础设施之后。

当前上线版本没有把隐私友好型好友与分享功能放进关键路径，已交付体验始终聚焦目标、每日进度、私人历史与宠物陪伴循环。
